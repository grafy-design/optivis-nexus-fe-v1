# Playwright Page Diff Review Guideline

## 목적

이 문서는 `3003` 목표 포트와 `3000` 작업 포트를 같은 페이지 기준으로 비교할 때 따르는 작업 규칙을 정리한다.

핵심 목표는 다음과 같다.

- 코드 수정 없이 화면 차이만 확인한다.
- `Playwright MCP`로 실제 렌더링 결과를 기준으로 비교한다.
- 페이지 주소와 설명만 읽어도 차이를 이해할 수 있을 정도로 구체적으로 기록한다.
- 기본적으로 차이점만 정리한다.

## 고정 기준

- 목표 포트: `3003`
- 작업 포트: `3000`
- 기준 방향: `3003`을 기준 화면으로 보고, `3000`이 어떻게 다른지 설명한다.
- 코드 수정: 하지 않는다.
- 비교 도구: `Playwright MCP`

## 비교 전 준비 규칙

비교 전에 아래 조건을 반드시 맞춘다.

1. 두 포트에서 같은 페이지를 연다.
2. 뷰포트를 동일하게 맞춘다.
3. 로딩이 끝난 뒤 비교한다.
4. 펼침/접힘 상태를 동일하게 맞춘다.
5. 선택 상태, 탭 상태, 스텝 상태가 다르면 가능한 한 동일하게 맞춘다.
6. URL 쿼리가 다르면 먼저 기록하고, 그 차이가 상태 차이로 이어질 수 있음을 함께 적는다.

## 로딩 완료 판정

아래 조건 중 하나라도 남아 있으면 바로 비교하지 않는다.

- `Loading...` 텍스트가 남아 있음
- 스켈레톤 UI가 남아 있음
- 그래프나 테이블 데이터가 아직 채워지지 않음
- 비동기 렌더링으로 카드 높이나 표 레이아웃이 계속 바뀌는 중임

필요하면 잠시 대기한 뒤 다시 확인한다.

## 상태 정렬 규칙

같은 페이지라도 상태가 다르면 화면 차이가 왜곡될 수 있으므로 아래를 맞춘다.

- 같은 행 확장 상태
- 같은 선택 라디오/체크 상태
- 같은 스텝 위치
- 같은 버튼 활성/비활성 상태 여부

상태를 맞출 수 없으면, 그 차이를 먼저 명시한 뒤 비교 결과에 포함한다.

## 필수 비교 항목

### 1. 레이아웃

- 전체 2단 또는 다단 구조가 같은지 확인
- 좌우 패널 비율이 같은지 확인
- 섹션의 시작 위치, 정렬 축, 카드 배치 순서 확인
- 확장 패널이 들어왔을 때 주변 영역이 밀리거나 깨지는지 확인

### 2. 간격

- 섹션 간 여백
- 카드 내부 패딩
- 제목과 본문 간격
- 표 헤더와 행 사이 간격
- 하단 버튼과 본문 사이 간격

### 3. 색상

- 배경색
- 텍스트 색상
- 강조 색상
- 활성/비활성 버튼 색상
- 경계선, 구분선, 그림자 강도

### 4. 아이콘

- 아이콘 종류가 같은지 확인
- 아이콘 크기와 두께 확인
- 아이콘 색상 확인
- 눌림/선택/비활성 상태가 같은지 확인

### 5. 컴포넌트 크기

- 카드 높이와 너비
- 버튼 높이와 너비
- 테이블 행 높이
- 그래프 영역 높이와 너비
- 입력 요소, 라디오, 체크박스, 액션 아이콘 크기

### 6. 폰트

아래 항목을 주요 텍스트 요소별로 확인한다.

- `font-family`
- `font-size`
- `font-weight`
- `line-height`
- `letter-spacing`

주요 텍스트 범위는 다음을 포함한다.

- 메인 타이틀
- 서브타이틀
- 스텝 네비게이션
- 카드 제목
- 테이블 헤더
- 테이블 본문
- 상세 카드 설명 텍스트
- 강조 수치 텍스트
- 하단 CTA 버튼 텍스트

### 7. 상태 차이

- 버튼 활성/비활성
- 선택 여부
- 확장/접힘 여부
- 강조 단계 표시
- 데이터 표시 여부

### 8. 기타 차이

아래처럼 체크리스트 밖의 차이도 발견하면 기록한다.

- 수치 값 차이
- 텍스트 줄바꿈 차이
- 말줄임 처리 차이
- 스크롤 발생 여부
- 잘림, 겹침, 오버플로우
- 그림자, 라운드, 보더 두께 차이

## 비교 방법

1. `3000`과 `3003`에서 같은 페이지를 연다.
2. 같은 뷰포트로 맞춘다.
3. 로딩 완료까지 기다린다.
4. 필요한 경우 같은 행을 펼친다.
5. 스크린샷으로 전체 레이아웃을 먼저 확인한다.
6. `computed style`과 `bounding box`로 주요 요소의 실제 수치를 확인한다.
7. 차이점만 정리한다.

## 보고 원칙

- 기본적으로 차이점만 적는다.
- 차이가 없으면 `차이 없음`이라고 명확히 적는다.
- `3003` 기준으로 `3000`이 어떻게 다른지 쓴다.
- 단순히 “다름”이라고 쓰지 말고, 사용자가 바로 떠올릴 수 있게 영역과 증상을 같이 적는다.
- 수치가 있으면 가능한 한 같이 적는다.
- 상태 차이와 스타일 차이는 구분해서 적는다.
- 로딩 중 상태가 섞였으면 그 비교는 무효로 보고 다시 확인한다.

## 권장 보고 형식

```md
페이지: /example/path
비교 기준: 3003 기준, 3000 차이점만 정리
비교 상태: 로딩 완료 / 1번 행 펼침 / 동일 뷰포트

- 하단 `Subgroup Explain` 버튼 상태가 다름.
  3003은 비활성 회색, 3000은 활성 오렌지.

- 우측 상세 카드의 강조 수치가 다름.
  3003은 `Variance 2.74`, `VR 0.070`.
  3000은 `Variance 0.78`, `VR 0.780`.

- 좌측 카드 제목 폰트 크기가 다름.
  3003은 `19.5px`, 3000은 `24px`.
```

## 금지 사항

- 화면 비교 작업 중 코드 수정 금지
- 로딩 중 화면을 기준으로 성급하게 결론 내리지 않기
- 상태가 다른 화면을 같은 화면처럼 단정하지 않기
- 색, 간격, 폰트 차이를 추측으로 적지 않기

## 최종 요약 규칙

최종 답변은 아래 원칙을 따른다.

- 장황한 설명보다 차이점 중심으로 정리
- 중요도가 높은 차이부터 먼저 정리
- 사용자가 바로 수정 포인트를 알 수 있도록 영역 중심으로 작성
- 필요하면 `레이아웃`, `폰트`, `상태`, `수치` 정도로만 짧게 구분
